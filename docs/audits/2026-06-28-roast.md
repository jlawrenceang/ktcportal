# KTC Online Portal — UX Roast (2026-06-28)

**Method:** `roast` lens. Real shipped app at `portal.ktcterminal.com` (v1.6.74), captured via a minted owner magic-link session (service-role `generateLink`), Playwright, desktop 1440×900 + mobile 390×844. 42 screenshots in scratchpad `roast-shots/`. STRICT READ-ONLY: navigate + screenshot only; no clicks, no form submits, no business-data writes.

**Site type:** mixed — (a) marketing-ish public landing/login + (b) gated B2B portal + (c) internal/ops back office. Each surface graded under its own type; no blended composite.

---

## ⚠️ Capture tier: PARTIAL (Tier 1) — the primary target was NOT reached

**The staff back office and staff PWA were unreachable, so the highest-value surfaces are UNSCORED.** Every staff route (`/admin`, `/admin/job-orders`, `/admin/cashier`, `/admin/checker`, `/admin/releases`, `/admin/consignees`, `/admin/approvals`, `/admin/settings`, `/admin/vessel-schedule`, `/admin/logs`, `/app`, `/app/checker`, `/app/cashier`) redirected to the **customer "Finish verifying your account" pending gate** (evidence: `desktop__admin-job-orders.png`, `desktop__admin-cashier.png`, `desktop__app.png` all render the same pending card).

**Root cause (from code, not inference):** the frontend derives admin access from `hasAdminAccess(broker)`, reading the `is_admin`/`is_owner` columns off the user's own `customers` row fetched by `useBroker` (`src/lib/useBroker.ts:47-51`, GET `.eq('user_id', uid).maybeSingle()`). The owner's magic-link session resolved to a **pending, non-admin** customer row → all staff routes redirect to the pending gate. This is NOT a capture artifact: `useBroker` uses a GET, which the capture passed through untouched. Likely either (a) the owner's `customers` row lacks `is_owner=true`/`is_admin=true`, or (b) duplicate rows for that `user_id` make `.maybeSingle()` return null (the exact failure the code comments warn about at `useBroker.ts:19`). DB verification was correctly denied under read-only, so this is left for the owner to confirm.

**Read-only capture notes (no business data touched):**
- Pre-set client-only flags to reveal real screens: `ktc_lang`/`ktc_lang_set` (skip language chooser), `ktc_setup_done_<uid>` (skip first-run "Quick setup"), `ktc_tour_shown_*` sessionStorage (skip auto tours). All localStorage/sessionStorage only.
- Single-session guard: the `has_other_live_session` **read** RPC was stubbed to `false` so the app took its normal clean-login path; `claim_session` (the required, routine login claim that every owner login performs) ran; any table-write verb was network-swallowed. No DELETE/eviction SQL was run (an attempt was correctly blocked by the safety classifier and abandoned).

**Customer-as-owner caveat:** the only reachable authed surface is the customer portal in a **pending** state, navigated **as the owner account** — not a true customer session and not an approved customer. Treat type-(b) findings as indicative.

## Verdict & FAIL line

- **Public landing/login (type a):** solid — clear value prop + single dominant CTA. Would not FAIL.
- **Gated portal pending gate (type b):** clean and legible, but only the locked verification state was visible.
- **Ops back office (type c):** **UNSCORED — not reached.** A type-(c) roast FAILs if density or efficiency accelerators are weak; that cannot be judged here.

Composite is a partial diagnostic, not a calibrated score.

## Dimension scores (cite a screen per claim)

### 1. Information hierarchy & scannability — 80 (reachable surfaces only)
- Public home (`desktop__public-home.png`): one H1 "KTC Online Portal", a one-line value prop ("The online service desk of KTC Container Terminal Corp — file and track your terminal and port-services work"), then a CTA stack (Sign in / Create an account / Continue with Google). Clean single focal path.
- Pending gate (`desktop__cust-job-orders.png`): exemplary — single H1 "Finish verifying your account", one explanation line, one dominant action "Upload valid ID", two secondary links. A user knows the next action instantly.
- Caveat: the dimension's core scenario (a busy staffer finding the next action in an ops queue) is **unseen**.

### 2. Density vs clarity on ops queues — UNSCORED
- The job-orders / cashier / checker queues were never reached (`desktop__admin-job-orders.png` = pending gate). No honest score possible.

### 3. Affordance & action clarity — 80
- Primary actions are unambiguous orange filled buttons ("Sign in" `desktop__public-login.png`; "Upload valid ID" `mobile__cust-job-orders.png`). State is legible: the amber pending banner ("orders can't be filed until a KTC admin approves your account", `desktop__cust-job-orders.png`) reads clearly.
- Caveat: queue-row chips/status badges (the real test of state legibility) are unseen.

### 4. Visual consistency — 85
- One accent (orange gradient) and consistent components across public, modals, and pending gate: same primary button, same card radius, same logo lockup (`desktop__public-login.png`, `desktop__public-verify-notfound.png`, `desktop__cust-calculator.png`).
- Minor inconsistency: redirected staff routes render the pending card over the **port-photo** background (`desktop__admin-job-orders.png`) while the customer route renders it over a **plain gradient** (`desktop__cust-job-orders.png`) — two backgrounds for the same card.

### 5. Feedback & states — 80
- Strong error state: invalid JO code → a clean "NOT FOUND" card with a plain-language explanation (`desktop__public-verify-notfound.png`).
- Pending/awaiting state is explicit (`mobile__cust-job-orders.png`); loading shown via the Turnstile "Verifying…" widget on login (`desktop__public-login.png`); single-session conflict has a real modal (captured earlier this run); first-run "Quick setup" modal exists.
- No raw/empty/crashing states observed on reachable screens. Caveat: ops-queue empty/loading states unseen.

### 6. Responsive + staff PWA — 85 (public/customer) · PWA UNSCORED
- Mobile renders cleanly: single column, generous tap targets, no horizontal overflow on home/login/register/pending (`mobile__public-home.png`, `mobile__public-login.png`, `mobile__cust-job-orders.png`).
- The focused staff PWA screens (`/app/*`) were **unreached** (redirect to pending gate) — the dimension's PWA half is unscored.

### 7. Distinctiveness / anti-slop — 68
- Distinctive and subject-rooted: a real aerial of the container terminal is the hero (`desktop__public-home.png`) — the data/subject is the hero, one orange accent. Not a generic template.
- Anti-slop deductions: heavy frosted-**glass** cards over the photo on all four public screens (home/login/register/agreement) — the framework prefers hairlines over glass/blur; login form fields sit on a translucent panel that softens label contrast (`desktop__public-login.png`). Authed surfaces fall back to a generic card-on-light-gradient that "could be any SaaS" (`desktop__cust-calculator.png`).

### 8. Onboarding / first impression — 75
- Public first impression is strong (clear value prop + CTAs, `desktop__public-home.png`).
- First authed run **stacks interruptions**: a full-screen "Quick setup" modal (language + notifications) then per-page auto-tours, each a full overlay, before any content (captured pre-suppression: `Quick setup` on first load; `WelcomeTour` steps on calculator/vessels). Each is clear, but it's two sequential gates before value.

## Prioritized fix list (highest impact first)

1. **[BLOCKER] Owner session cannot reach the staff back office.** `desktop__admin-job-orders.png` — owner magic-link resolves to a pending non-admin customer; all staff routes redirect to the verification gate. Verify the owner's `customers` row has `is_owner/is_admin=true` and is unique per `user_id` (duplicate rows → `useBroker.maybeSingle()` returns null → no admin UI). Consider falling back to the server `is_owner()` failsafe in `hasAdminAccess`, not only the column. **Effort: M** (data check S; code hardening M).
2. **[HIGH] Re-run this roast against a real staff/owner-admin session** so the ops queues (job-orders, cashier, checker) and staff PWA get graded — they are the product's core and are currently unscored. `desktop__admin-*` / `desktop__app-*`. **Effort: S** (once #1 is resolved).
3. **[MED] Reduce first-run modal stacking.** First load fires "Quick setup" then per-page tours as separate full-screen overlays (`Quick setup` first-load capture; `WelcomeTour` on `cust-calculator`). Fold setup into a single dismissible step and make tours opt-in (a "?" pulse) rather than auto-opening on every page. **Effort: M**.
4. **[MED] De-glass the public auth screens.** `desktop__public-login.png` — form labels/fields sit on a frosted panel over the photo, softening contrast. Swap to a solid panel with a hairline border (keep the photo as a side/background band) to raise text contrast and de-slop. **Effort: M**.
5. **[MED] Tighten the mobile register screen.** `mobile__public-register.png` — fields + inline agreement scroll-box + consent are cramped in one tight column; the agreement box is very short on mobile. Give the agreement more height or a "read agreement" expand, and space the fields. **Effort: M**.
6. **[LOW] Raise contrast of secondary text over the photo.** `mobile__public-home.png` — the small address line under the logo and the "Secure · SSL-encrypted" footer are small low-contrast grey on the image. Bump weight/contrast or place on a solid chip. **Effort: S**.
7. **[LOW] Improve information scent of the home accordions.** `desktop__public-home.png` — the four collapsed rows (Job Orders / Container Release & Pull-out / Online Payments / Vessel Schedule & Rates) reveal nothing until expanded. Show a one-line subtitle per row so the value is legible at a glance. **Effort: S**.
8. **[LOW] Unify the pending-gate background.** Same card renders over a port photo on staff routes (`desktop__admin-job-orders.png`) and a plain gradient on the customer route (`desktop__cust-job-orders.png`). Pick one. Also fix doc drift: `CLAUDE.md` still says "pending brokers get the full portal and can file job orders," but the shipped gate locks the portal ("orders can't be filed until a KTC admin approves") — align the doc to the current verify-only lifecycle. **Effort: S**.

## Closing lens
Would a sharp engineer ship the *reachable* screens? Yes — the public landing, the NOT-FOUND state, and the pending gate are clean, consistent, and identity-rooted. But the roast that matters — the dense ops queues a checker/cashier lives in all day — could not be reached because the owner session renders as a customer. **Resolve fix #1, then re-run for the real verdict.**
